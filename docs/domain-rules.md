# Domain Rules

## General Rules

### Request Identity
- every OfficeAdmin task request must have a unique request ID
- request IDs are UUID4 strings, generated by OfficeAdmin at submission time
- request IDs are used by the backend and UI to query status and cancel work
- format: lowercase hyphenated UUID string, e.g. `"3f2504e0-4f89-11d3-9a0c-0305e82c3301"`

### Date Format
- all `selected_date` values are ISO 8601 date strings: `YYYY-MM-DD`, e.g. `"2026-04-10"`
- any other format is rejected with HTTP 400
- the backend validates this before passing to OfficeAdmin

### Timezone Handling
- `selected_date` represents a full calendar day without timezone qualification
- CalendarWorker queries Google Calendar using the **local system timezone**, not UTC: `timeMin` is local midnight on the selected date; `timeMax` is local midnight on the following date
- only timed events (having `start.dateTime`) that start at or after 08:00 local and end at or before 18:00 local are included; all-day events are excluded
- v1 targets **desktop-only deployment** (macOS/Windows); the local system timezone is the user's own timezone, which is correct for this use case
- containerized or server deployments would need an explicit `TZ` environment variable; this is out of scope for v1
- per-user timezone support is out of scope for v1

### Message-Based Processing
- public task-start methods enqueue work and return immediately
- component-to-component communication is asyncio-based and non-blocking
- all components run within the same asyncio event loop
- blocking work (HTTP calls, PDF generation, printing, OAuth) runs in an executor via `run_in_executor`

---

## Concurrency Rules

### asyncio Model
- all components use `asyncio.Queue` and `asyncio.Task`
- no `threading.Thread` or `threading.Lock` is used
- because asyncio is cooperative, state mutations that do not cross an `await` boundary are safe without locks
- compound check-then-act operations must complete before any `await`

### OfficeAdmin Queue Limit
- OfficeAdmin's work queue has a maximum depth of 10
- if a new task is submitted when the queue is at capacity, the submission is rejected
- FastAPI returns HTTP 429 when this occurs

## Cancellation Rules

### Requesting Cancellation
- a user may request cancellation of any task that is not yet in a terminal state
- terminal states are: COMPLETED, CANCELLED, ERROR
- OfficeAdmin sets `cancel_requested = True` and `status = CANCEL_REQUESTED`
- OfficeAdmin propagates `cancel_request(request_id)` to all active workers

### Worker Cancellation Callback Contract
- when a worker observes cancellation (at work-item start or before callback), it must call the `*_failed` callback with `error_text = "Cancelled"`
- workers must never silently drop a work item; every item must produce exactly one callback
- this ensures OfficeAdmin's in-flight counters always reach their expected totals

### Cancellation Finalization
- OfficeAdmin determines the nature of each failure callback by checking `cancel_requested`:
  - if `cancel_requested = True`: treat as a cancellation failure; count toward total; do not set ERROR
  - if `cancel_requested = False`: treat as a real error; see Failure Rules below
- when all expected callbacks have been received and `cancel_requested = True`: set `status = CANCELLED`, `stage = CANCELLED`
  - for PRINT_CALENDAR_EVENTS: `documents_completed + documents_failed == documents_expected` (at document stage) or `prints_completed + prints_failed == prints_expected` (at print stage)
  - for SEND_EMAIL_NOTIFICATIONS: `emails_completed + emails_skipped + emails_failed == emails_expected`; **skipped events count toward the total** — an all-skipped day finishes normally, not by hanging
- idempotent: calling `cancel_request` on an already-cancelled or terminal task is a no-op

### Late-Arriving Callbacks
- if a callback arrives for a task whose status is already terminal (COMPLETED, CANCELLED, ERROR), it is discarded silently

---

## Failure Rules

### Partial Failure Policy
- the first non-cancellation failure from any worker immediately terminates the entire task with ERROR
- OfficeAdmin sets `status = ERROR`, `stage = ERROR`
- OfficeAdmin propagates `cancel_request(request_id)` to all workers to halt remaining in-flight work
- already-completed subtask results are recorded in the status entry but not acted on
- no partial printing occurs: if document generation fails, no print jobs are dispatched

### Worker Crash Policy (v1)
- if a worker's asyncio task terminates due to an unhandled exception, the exception is logged
- affected tasks remain in their last known status
- automated recovery is out of scope for v1

## Office Task Rules

### Supported Task Types
- `PRINT_CALENDAR_EVENTS`
- `SEND_EMAIL_NOTIFICATIONS`

### Print Calendar Events Task Input
Required:
- `selected_date` (ISO 8601 date string)

### Calendar Widget Default
- for **Print Calendar Events**: the date picker defaults to today (you are printing today's events)
- for **Send Email Notifications**: the date picker defaults to tomorrow (you are notifying for upcoming events)

## Send Email Notifications Rules

### Draft Creation
- one Gmail draft is created per calendar event

### Recipient Extraction
- recipients are extracted ONLY from `event.description`
- extraction scans the full description text for email address patterns
- not limited to `mailto:` links

### Multiple Recipients
- if multiple email addresses exist in the description:
  - include all in the same draft

### No Recipients
- if no email addresses are found:
  - skip event
  - do not create draft

### Duplicate Emails
- duplicate addresses must be removed

### Email Behavior
- drafts only (no sending)

## Task Status Rules

### Top-Level Task Status Values
| Value | Description |
|---|---|
| `PENDING` | Submitted, not yet started |
| `RUNNING` | Actively processing |
| `COMPLETED` | Finished successfully |
| `CANCEL_REQUESTED` | Cancellation requested; propagating to workers |
| `CANCELLED` | All in-flight work reported back; fully cancelled |
| `ERROR` | A non-cancellation failure occurred |
| `UNKNOWN` | Request ID not found |

### Stage Values for Print Calendar Events
| Value | Description |
|---|---|
| `PENDING` | Not yet started |
| `GETTING_CALENDAR_EVENTS` | CalendarWorker retrieving events |
| `CREATING_EVENT_PDFS` | DocumentWorker generating PDFs |
| `PRINTING_EVENT_PDFS` | PrinterWorker printing PDFs |
| `COMPLETED` | Workflow finished successfully |
| `CANCELLED` | Workflow cancelled |
| `ERROR` | Workflow failed |

### Stage Values for Send Email Notifications
| Value | Description |
|---|---|
| `PENDING` | Not yet started |
| `GETTING_CALENDAR_EVENTS` | CalendarWorker retrieving events |
| `CREATING_EMAIL_DRAFTS` | MailWorker creating Gmail drafts |
| `COMPLETED` | Workflow finished successfully |
| `CANCELLED` | Workflow cancelled |
| `ERROR` | Workflow failed |

### Valid (status, stage) Pairs
| status | stage | task types |
|---|---|---|
| PENDING | PENDING | both |
| RUNNING | GETTING_CALENDAR_EVENTS | both |
| RUNNING | CREATING_EVENT_PDFS | PRINT_CALENDAR_EVENTS |
| RUNNING | PRINTING_EVENT_PDFS | PRINT_CALENDAR_EVENTS |
| RUNNING | CREATING_EMAIL_DRAFTS | SEND_EMAIL_NOTIFICATIONS |
| CANCEL_REQUESTED | GETTING_CALENDAR_EVENTS | both |
| CANCEL_REQUESTED | CREATING_EVENT_PDFS | PRINT_CALENDAR_EVENTS |
| CANCEL_REQUESTED | PRINTING_EVENT_PDFS | PRINT_CALENDAR_EVENTS |
| CANCEL_REQUESTED | CREATING_EMAIL_DRAFTS | SEND_EMAIL_NOTIFICATIONS |
| COMPLETED | COMPLETED | both |
| CANCELLED | CANCELLED | both |
| ERROR | ERROR | both |

### Status Entry Shape

All requests include:
- `request_id`
- `task_type`
- `status`
- `stage`
- `selected_date`
- `calendar_event_count`
- `events_retrieved`
- `cancel_requested`
- `errors`
- `created_at`
- `updated_at`

For `PRINT_CALENDAR_EVENTS` requests, also include:
- `documents_expected`
- `documents_completed`
- `documents_failed`
- `prints_expected`
- `prints_completed`
- `prints_failed`
- `document_paths`

For `SEND_EMAIL_NOTIFICATIONS` requests, also include:
- `emails_expected`
- `emails_completed`
- `emails_skipped`
- `emails_failed`
- `draft_ids`
- `skipped_event_ids`

### Unknown Request Handling
- if a request ID is not found, status must be `UNKNOWN`

### Task Store Persistence
- the task store is in-memory only in v1
- all task state is lost on server restart
- this is documented behavior
- tasks are removed from the in-memory store 30 minutes after reaching a terminal state (COMPLETED, CANCELLED, ERROR); polling after that window returns UNKNOWN
- worker `cancelled` flags are removed 60 minutes after `cancel_request` is called to bound memory growth

## OfficeAdmin Rules

### Task Submission
When a print-calendar-events task is submitted, OfficeAdmin must:
1. validate the queue is not full; if full, raise an error (FastAPI returns HTTP 429)
2. create a UUID4 request ID
3. create a status entry with:
   - `task_type = "PRINT_CALENDAR_EVENTS"`
   - `status = "PENDING"`
   - `stage = "PENDING"`
   - `selected_date`
   - `cancel_requested = False`
4. enqueue a work item
5. return the request ID immediately

### Calendar Retrieval Dispatch
When OfficeAdmin begins processing a print-calendar-events request:
1. set `status = RUNNING`, `stage = GETTING_CALENDAR_EVENTS`
2. dispatch to CalendarWorker asynchronously

### Calendar Completion Handling
When CalendarWorker calls `calendar_events_complete`:
1. discard if task is terminal
2. verify task is not cancelled before dispatching new work
3. record retrieved event count
4. if zero events: set `status = COMPLETED`, `stage = COMPLETED`
5. otherwise: set `stage = CREATING_EVENT_PDFS`, set `documents_expected = N`, dispatch N document jobs to DocumentWorker

### Document Completion Handling
When DocumentWorker calls `document_complete`:
1. discard if task is terminal
2. record the document path, increment `documents_completed`
3. if `documents_completed == documents_expected`: verify not cancelled; set `stage = PRINTING_EVENT_PDFS`, set `prints_expected = N`, dispatch N print jobs to PrinterWorker

### Printer Completion Handling
When PrinterWorker calls `print_complete`:
1. discard if task is terminal
2. increment `prints_completed`
3. if `prints_completed == prints_expected`: set `status = COMPLETED`, `stage = COMPLETED`

### Failure Handling
When any `*_failed` callback is received:
1. discard if task is terminal
2. if `cancel_requested = True`: count as cancellation failure; if (completed + failed == expected for the active stage), set `status = CANCELLED`, `stage = CANCELLED`
3. if `cancel_requested = False`: set `status = ERROR`, `stage = ERROR`; propagate `cancel_request` to all workers; record error details

### Shutdown
OfficeAdmin must implement `async shutdown()` which enqueues a stop sentinel and awaits the worker task.

## CalendarWorker Rules

### Calendar Scope
- initial version uses the authenticated user's primary Google Calendar

### Event Retrieval Rules
- CalendarWorker queries events using local timezone boundaries (midnight-to-midnight local time on the selected date)
- only timed events (with `start.dateTime`) that start ≥ 08:00 local and end ≤ 18:00 local are returned
- all-day events are always excluded
- returned events are normalized into a consistent structure (see Specs/calendar-worker.md)

### Google OAuth
- CalendarWorker handles OAuth inline on the first request
- if no valid token exists, the OAuth flow is run in an executor (blocking)
- if OAuth fails, `calendar_events_failed` is called with the auth error
- once a token is obtained, it is persisted (e.g., `token.json`) and reused
- token refresh is handled transparently by the Google auth library

### Event Normalization
Each normalized event must include, when available:
- `id`
- `summary`
- `start`
- `end`
- `location`
- `description`
- `html_link`
- `status`

### Cancellation
- CalendarWorker checks `cancel_requested` at the start of each work item
- if cancelled: calls `calendar_events_failed(request_id, selected_date, "Cancelled")`
- CalendarWorker must implement `async shutdown()`

---

## DocumentWorker Rules

### One PDF Per Event
- DocumentWorker creates exactly one PDF per calendar event
- each PDF corresponds to exactly one event

### PDF Content
Each PDF must include all available event details:
- summary
- date
- start and end time
- location
- description
- event status
- html link (if present)

### Output Location
- PDFs are written to the project `reports/` directory
- output directory must be configurable (for test redirection)

### File Naming
File names are deterministic, safe for the filesystem, and collision-aware.
Required components:
- request ID prefix (first 8 characters)
- event ID or safe event index
- truncated safe summary fragment (alphanumeric only, max 40 chars)
- `.pdf` extension

Example: `3f2504e0_evt001_team-standup.pdf`

### Cancellation
- DocumentWorker checks `cancel_requested` at the start of each work item
- if cancelled: calls `document_failed(request_id, event_id, "Cancelled")`
- DocumentWorker must implement `async shutdown()`

---

## PrinterWorker Rules

### Print Inputs
- PrinterWorker receives document paths for PDFs created by DocumentWorker

### Printing
- PrinterWorker attempts to print the provided PDFs via the system default printer in v1
- print integration is isolated behind a helper/adapter for testability
- print submission runs in an executor

### Cancellation
- PrinterWorker checks `cancel_requested` before starting each print job
- if cancelled: calls `print_failed(request_id, event_id, "Cancelled")`
- if a print is already submitted to the OS, cancellation is best-effort
- PrinterWorker must implement `async shutdown()`

## MailWorker Rules

### Recipient Extraction
- recipients are extracted from `event.description` only; other event fields are not searched
- the extraction pattern matches email addresses found anywhere in the text
- multiple addresses in one description yield multiple recipients
- duplicate addresses are removed; first occurrence is kept
- events with no email addresses → `email_draft_skipped` callback (not `email_draft_failed`)

### Draft Content Template
- email content is loaded from `templates/email_notification_template` at compose time
- the first line of the template is the email subject (stripped of whitespace)
- the remaining text is the email body
- the body supports three placeholders:
  - `{date}` — full local date of the event start (e.g., `Saturday, April 12, 2026`)
  - `{time}` — local start–end time range (e.g., `9:00 AM – 10:00 AM`); `(all day)` for all-day events
  - `{location}` — value of `event["location"]`; empty string if `None`
- the template file must exist on disk; missing file is a runtime error (not silently ignored)
- the template is not hardcoded in source code; updating the file changes all future drafts without a code deployment

### Draft Creation
- one Gmail draft per event (for events with at least one recipient)
- drafts are created via the Gmail API; no email is sent
- if draft creation fails: `email_draft_failed` callback with error text

### Finalization Counter
- every event dispatched to MailWorker must produce exactly one callback: `complete`, `skipped`, or `failed`
- `emails_expected = number of events dispatched`
- `emails_completed + emails_skipped + emails_failed == emails_expected` triggers finalization
- `skipped` is a normal outcome and counts toward the total

### Gmail OAuth
- MailWorker handles Gmail OAuth separately from CalendarWorker OAuth
- Gmail credentials: `gmail_credentials.json`, Gmail token: `gmail_token.json`
- required scope: `https://www.googleapis.com/auth/gmail.compose` (draft creation only)
- OAuth failure → `email_draft_failed` with auth error

### Cancellation
- MailWorker checks `cancel_requested` at the start of each work item and before any callback
- if cancelled: calls `email_draft_failed(request_id, event_id, "Cancelled")`
- MailWorker must implement `async shutdown()`

## FastAPI Rules

### Backend Responsibilities
- backend routes delegate to OfficeAdmin
- backend must not duplicate orchestration logic

### Lifespan
- on startup: create all components in order (CalendarWorker, DocumentWorker, PrinterWorker, MailWorker, OfficeAdmin)
- on shutdown: shut down in reverse order — OfficeAdmin first, then MailWorker, PrinterWorker, DocumentWorker, CalendarWorker

### Required Endpoints
- `POST /api/office/print-calendar-events` — start print task; returns request ID
- `POST /api/office/send-email-notifications` — start email task; returns request ID
- `GET /api/office/status/{request_id}` — get task status
- `POST /api/office/cancel/{request_id}` — request cancellation
- `GET /api/office/tasks` (optional) — list active/recent tasks

### Validation and Error Responses
| Condition | HTTP Response |
|---|---|
| Invalid date format | 400 Bad Request |
| Unknown request ID | 404 Not Found (status: UNKNOWN) |
| Queue full | 429 Too Many Requests |
| Internal failure | 500 Internal Server Error |

## Frontend Rules

### Required UI Behavior
- provide a `Print Calendar Events` button (date picker defaults to today)
- provide a `Send Email Notifications` button (date picker defaults to tomorrow)
- show a calendar/date widget when each button is pressed
- submit the selected date to the backend
- display running tasks with current stage
- allow cancellation of in-progress tasks

### Status Polling
- poll `/api/office/status/{request_id}` periodically for active tasks
- stop polling when the task reaches a terminal state

### Task Visibility
The UI must show:
- request ID
- task type
- selected date
- current stage (human-readable)
- top-level status
- progress details when available
- cancel button when task is not terminal

## Extensibility Rules

### New Task Types
New OfficeAdmin task types must define:
- input payload
- stage model and valid (status, stage) pairs
- success criteria
- cancellation behavior
- failure policy
- status payload shape
- shutdown behavior

### New Worker Component
New worker components must:
- use asyncio.Queue and asyncio.Task
- implement `cancel_request(request_id)`
- implement `async shutdown()`
- call exactly one callback per work item (complete, failed with error, or failed with "Cancelled")
